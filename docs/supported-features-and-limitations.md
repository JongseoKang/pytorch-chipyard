# Supported Features and Limitations

This page summarizes the current support boundary and known limitations of
PyTorch-Chipyard and Triton-Chipyard.

## Main Supported Path

The strongest documented path is:

```text
PyTorch model
  -> torch.compile / TorchInductor
  -> Triton-Chipyard backend
  -> Chipyard runner artifact
  -> Chipyard / FireSim target execution
```

The current support scope includes:

- Most Triton-style loopwise operator lowering generated by TorchInductor.
- Inductor template-based candidate generation for matmul, convolution, BMM, and
  attention-style kernels.
- Artifact generation for Gemmini, RVV, and scalar RISC-V backends.
- CNN examples: AlexNet, MobileNetV2, ResNet50, and SqueezeNet.
- LLM prefill examples: GPT-2, GPT-Neo, OPT, and Pythia.
- SDPA, FlashAttention-style, and windowed FlexAttention prefill artifact
  generation scripts.
- Generation of `runner.cpp`, `model_spec.json`, `weights.bin`,
  `weights.manifest.json`, `input.bin`, `output.bin`, `build.sh`, and `util.py`.
- Standalone Triton-Chipyard examples for matmul, convolution, and
  layernorm-style kernels.

## Artifact Contract

The generated artifact directory is the interface between PyTorch compilation
and Chipyard execution.

| File | Role |
| --- | --- |
| `runner.cpp` | Generated C++ runner containing kernel launch order, buffer allocation, input/output movement, and deferred autotune execution. |
| `model_spec.json` | Contract containing graph input, weight, buffer, output, kernel, size, stride, storage offset, and autotune site metadata. |
| `weights.bin` | Static model parameter and buffer payload. |
| `weights.manifest.json` | Metadata for tensors packed into `weights.bin`. |
| `input.bin` | Runtime input payload. |
| `output.bin` | Output payload expected to be written by target execution and read by validation. |
| `model.log` | Runtime model-cycle summary written by the generated runner in the target working directory. This file is produced after the runner executes, not during compilation. |
| `autotune.log` | Runtime autotune candidate and site-result log written by the generated runner in the target working directory. This file is produced after the runner executes, not during compilation. |
| `build.sh` | Script that builds `runner.cpp` and kernel objects into `model.elf`. In the documented direct flow this script is generated but not automatically called; run it explicitly from the artifact directory. |
| `util.py` | Helper for writing Python tensors to `input.bin` and reading `output.bin` back as PyTorch tensors. |

When debugging mismatches, use `model_spec.json` as the reference. If
`runner.cpp`, `weights.manifest.json`, and `util.py` disagree, first check the
tensor classification and buffer plan in `model_spec.json`.

The direct compilation flow uses:

```bash
python examples/ResNet50.py --compile --batch-size 1 --seed 0
```

Then build the target ELF explicitly:

```bash
cd "$PYTORCH_CHIPYARD_DUMP_PATH"
CHIPYARD_OMP_NUM_THREADS=4 bash ./build.sh
```

The resulting `model.elf` reads `input.bin` and `weights.bin` and writes
`output.bin`, `model.log`, and `autotune.log` when it is run on the target:

```bash
./model.elf input.bin weights.bin output.bin
```

FPGA setup, FireSim runtime setup, and FPGA execution are outside the coverage
of this work. Prepare and run the FPGA environment yourself, then copy the
result files back to the artifact directory for validation and analysis.

## Autotune Behavior

Triton-Chipyard does not provide exactly the same autotune behavior as the
standard Triton Python API. Standard Triton autotune caches kernel names,
arguments, and metadata on the host, then reuses the best kernel for later inputs
with the same shape without recompiling.

PyTorch-Chipyard can use Verilator-based cycle-accurate measurement, but that
path is very slow. For that reason, the host Python-level Triton autotune API is
not supported as-is.

The Inductor path differs in these ways:

- Simple loopwise kernels such as pointwise, reduction, and add kernels may be
  generated through the usual Inductor/Triton path, but PyTorch-Chipyard does
  not perform cycle-accurate autotuning for those kernels.
- Operators where Inductor templates generate multiple kernel candidates, such
  as matmul, convolution, and attention, record candidate metadata in
  `runner.cpp` and `model_spec.json`.
- The generated runner is structured so those deferred autotune sites can be
  evaluated through the target/cycle measurement path.
- `TORCHINDUCTOR_GEMMINI_MAX_AUTOTUNE=1` generates more Gemmini tiling
  candidates, but can increase simulation time significantly.

## `tl.dot` and Batched Matmul

Standard Triton can interpret rank-3 inputs to `tl.dot` as batched matmul.
Triton-Chipyard currently centers `tl.dot` support on 2D matmul forms.

When batched matmul is needed, use a Triton kernel template that handles the
batch dimension separately instead of relying on a single rank-3 `tl.dot`.
Inductor's Chipyard BMM template follows this representation.

## Graph Breaks

The current architecture adds the Triton-Chipyard backend and artifact generator
behind Inductor. One Inductor compile invocation produces one `runner.cpp` and
one artifact directory.

Models with graph breaks are therefore not yet supported as a single
end-to-end PyTorch-Chipyard artifact. A representative example is the decoding
step in Hugging Face models. Decoding commonly contains pure Python `if`
statements or loops that cause TorchDynamo/Inductor to be invoked multiple
times. The current artifact model does not support that kind of multi-graph
runtime orchestration.

The documented LLM path is centered on prefill.

## Target and Version Assumptions

This repository assumes the following custom checkout combination:

- Custom PyTorch with PyTorch-Chipyard Inductor integration.
- Custom Triton with out-of-tree Triton-Chipyard backend support.
- The `triton_chipyard` backend checkout.
- A matching LLVM/MLIR build.
- A matching Buddy-MLIR build.
- A Chipyard/Gemmini environment.

In particular, PyTorch, LLVM, Buddy-MLIR, and Triton-Chipyard should be used at
the versions included by this repository. Chipyard/FireSim paper artifact
replication also assumes the pinned commit and prebuilt bitstream combination.

## Current Limitations

Known current limitations:

- Triton-Chipyard is not a drop-in replacement for the full Triton Python API.
  The main targets are Inductor-generated kernels and the documented backend
  examples.
- Host-level Triton autotune cache/reuse semantics are not supported.
- Pointwise kernels do not receive cycle-accurate autotuning.
- `tl.dot` support is centered on 2D matmul. Rank-3 `tl.dot`-based batch matmul
  lowering is outside the current support boundary.
- Models with graph breaks cannot be packaged as one artifact.
- Standalone Verilator kernel execution can provide accurate cycle measurement,
  but it is very slow.
- FireSim/FPGA execution requires FPGA host setup, XRT/XDMA, Vivado/Vitis-
  compatible tooling, FireSim database files, and bitstreams in addition to the
  compiler stack.
- Some math rewrites and attention paths still require model- and shape-specific
  validation.
