# Section 4: Supported Features and Limitations

This section summarizes the current support boundary of PyTorch-Chipyard and the
main differences from upstream Triton behavior.

## Supported Operator Scope

Triton-Chipyard can lower most operators produced by TorchInductor's lowering
pipeline. The main supported path is still model-level PyTorch compilation
through TorchInductor:

```text
PyTorch model
  -> torch.compile / TorchInductor
  -> Triton-Chipyard backend
  -> Chipyard runner artifact
```

The intended support scope includes:

- Loopwise Triton kernels generated by Inductor for pointwise and reduction-like
  operators.
- Inductor template kernels for matmul, convolution, BMM, SDPA, and
  attention-related workloads.
- Model artifact generation for the documented CNN and LLM prefill examples.
- Gemmini, RVV, and scalar RISC-V lowering paths, depending on the selected
  environment variables.

## Difference from Triton Autotune

Triton-Chipyard does not support the standard Triton Python autotune API in the
same way as upstream Triton.

In upstream Triton, autotune caches the kernel name, arguments, metadata, and
best configuration on the host. When the same shape appears again, Triton can
reuse the best kernel without compiling and benchmarking every candidate again.

PyTorch-Chipyard can obtain cycle-accurate measurements through Verilator, but
that path is too slow to use as a general host-side Triton autotune mechanism.
As a result, host-level Triton autotune is not supported.

The practical consequence in the Inductor path is:

- For relatively simple pointwise and loopwise kernels, such as reductions and
  adds, Inductor may generate Triton kernels through its normal path, but
  PyTorch-Chipyard does not perform cycle-accurate autotuning for those kernels.
- For template kernels such as matmul, convolution, and attention, Inductor
  creates multiple kernel candidates and manages autotune metadata at the
  operator/template level.
- PyTorch-Chipyard records those operator-level autotune candidates in
  `runner.cpp` and the model artifact metadata so that the generated runner can
  perform target-cycle-based selection.

## `tl.dot` and Batched Matmul

Upstream Triton can interpret rank-3 inputs to `tl.dot` as batched matmul and
lower them accordingly.

Triton-Chipyard currently accepts `tl.dot` as a 2D matmul form. When batched
matmul is needed, the batch dimension should be represented explicitly at the
Triton kernel level rather than as a rank-3 `tl.dot`. The Inductor BMM template
used by PyTorch-Chipyard follows this structure.

## Graph Breaks

PyTorch-Chipyard is implemented as a backend path behind TorchInductor. One
Inductor invocation produces one `runner.cpp` and one model artifact directory.

Models with graph breaks are therefore not currently supported as one
end-to-end PyTorch-Chipyard artifact. A representative example is the decoding
step in many Hugging Face models. Decoding paths often contain pure Python
control flow, such as `if` statements or loops, which cause TorchDynamo and
Inductor to be invoked multiple times. The current runner-generation structure
does not orchestrate multiple separate Inductor graphs as one model artifact.

The documented LLM path is prefill-oriented rather than decoding-oriented.
